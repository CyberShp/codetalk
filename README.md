# CodeTalk

CodeTalk 是面向研发、测试、架构和代码审查团队的代码理解与分析工作台。它把代码仓库、辅助分析工具、LLM 配置、工作空间和分析结果集中到一个界面里，让团队成员可以更快地理解陌生代码、整理分析材料、生成报告和追踪问题线索。

这份 README 面向 CodeTalk 的使用者，而不是开发 CodeTalk 的工程团队。

## 适合谁使用

CodeTalk 适合这些场景：

- 新成员快速理解一个代码仓库。
- 测试同事围绕某个功能、模块或风险点准备分析材料。
- 研发同事梳理调用关系、模块边界、异常路径和关键流程。
- 架构或评审同事把代码证据整理成可讨论的报告。
- 团队需要把 GitNexus、CGC 和本机 Agent 等工具的结果放到同一个工作流里使用。

你不需要理解 CodeTalk 的内部实现，也不需要手动调用后端 API。正常使用从浏览器界面开始。

## 快速开始

### 1. 启动部署器

在 CodeTalk 目录下进入 `deployer`：

```powershell
cd deployer
.\start.bat
```

macOS/Linux:

```bash
cd deployer
./start.sh
```

也可以直接运行 `python start.py`。启动脚本会自动创建部署器 `.venv` 并安装依赖，避免因为缺少 `psutil`、`uvicorn` 等包导致部署系统起不来。

浏览器打开：

```text
http://127.0.0.1:9000
```

如果团队已经有人帮你部署好了 CodeTalk，直接打开他们给你的 CodeTalk 地址即可，不需要执行上面的命令。

### 2. 完成部署向导

在部署器首页点击部署入口，按页面提示完成：

| 步骤 | 你需要做什么 |
|---|---|
| 选择模式 | 普通本机使用选择 `native` |
| 环境检查 | 等待 Python、Node.js、Git、端口等检查完成 |
| 组件选择 | 不确定时先关闭 GitNexus / CGC 等可选组件 |
| 端口配置 | 默认可用即可；端口冲突时在高级设置里改端口 |
| 配置确认 | 检查前端、后端、CORS 和工作目录配置 |

当前核心功能只要求 backend + frontend 可用。GitNexus / CGC / 本机 Agent 是增强组件，不影响你先使用基础工作空间能力。

### 3. 启动 CodeTalk

部署完成后进入启动页，点击：

```text
Start All
```

页面上 backend 和 frontend 都显示运行后，点击：

```text
Open CodeTalk
```

默认地址通常是：

```text
http://127.0.0.1:3003
```

如果部署时改过端口，以启动页显示的地址为准。

## 主要工作流

### 创建工作空间

1. 打开 CodeTalk。
2. 进入“工作空间”。
3. 点击“新建工作空间”。
4. 填写工作空间名称。
5. 填写本机代码仓库路径。
6. 提交后进入工作空间详情页。

仓库路径应填写你机器上真实存在的目录，例如：

```text
E:\projects\my-service
```

或者：

```text
/home/user/projects/my-service
```

### 查看工作空间

工作空间详情页会展示当前仓库的索引、材料、报告、聊天和分析入口。不同环境下可用能力可能不同：

| 区域 | 用途 |
|---|---|
| 概览 | 查看当前工作空间状态 |
| 材料 | 添加设计文档、需求说明、测试材料等辅助信息 |
| 报告 | 查看或导出已经生成的分析结果 |
| 聊天 | 围绕当前工作空间提问 |
| 设置/状态 | 查看索引、模型和工具状态 |

如果某个增强工具未安装，相关区域可能显示不可用或降级提示，这是正常情况。

### 配置 LLM

进入“设置”页，配置团队可用的模型服务：

| 配置项 | 说明 |
|---|---|
| API 类型 | OpenAI compatible、Anthropic、Google、Ollama 等 |
| Base URL | 团队模型网关或本地模型地址 |
| API Key | 对应模型服务的访问密钥 |
| 模型名 | 例如 `gpt-4.1`、`qwen`、`claude` 或团队内部模型名 |

配置完成后，可以使用页面上的测试按钮确认模型是否可连通。

### 使用可选组件

GitNexus、CGC 和本机 Agent 属于增强能力：

| 组件 | 能力 | 是否必须 |
|---|---|---|
| GitNexus | 代码图谱、调用关系、结构化证据增强 | 否 |
| CGC | 更深的代码分析能力 | 否 |
| 本机 Agent | 通过 Claude Code Router、OpenCode、NGA 等执行智能体编排 | 否 |

如果你只是创建工作空间、查看基础页面、配置 LLM、整理材料，可以先不启用这些组件。

需要启用时，请在部署器或设置页里配置对应组件的路径、端口和命令。组件不可用时，CodeTalk 应该给出降级或错误提示，而不是拖垮核心页面。

## 常见问题

### 打不开 CodeTalk 首页

先确认启动页里 frontend 是否显示运行。

如果端口冲突，在部署器高级设置里换一个 frontend 端口，然后重新启动。

### 后端不可用

启动页里 backend 应显示运行。也可以在浏览器访问：

```text
http://127.0.0.1:3004/health
```

正常情况下会看到：

```json
{"status":"ok"}
```

如果端口不是 `3004`，以部署器启动页显示的 backend 地址为准。

### 新建工作空间失败

优先检查：

- 仓库路径是否真实存在。
- 路径是否填到了具体仓库目录，而不是不存在的父目录。
- 当前用户是否有读取该目录的权限。
- 后端是否正在运行。

### LLM 测试失败

检查：

- Base URL 是否能从当前机器访问。
- API Key 是否正确。
- 模型名是否存在。
- 团队模型网关是否要求额外的网络或代理配置。

### GitNexus / CGC / 本机 Agent 不可用

这些组件不是基础使用的必需项。你可以先关闭它们使用核心功能。

如果确实需要，请确认：

- 组件已经安装。
- 配置路径正确。
- 端口未被占用。
- health/status 检查通过。

## 使用建议

第一次使用时，建议按这个顺序：

1. 先启动 core：backend + frontend。
2. 创建一个小型代码仓库的工作空间。
3. 确认工作空间列表、新建页、详情页和设置页都能打开。
4. 配置 LLM。
5. 再按需启用 GitNexus / CGC / 本机 Agent。

这样可以把“CodeTalk 核心是否可用”和“增强组件是否配置好”分开判断，排障会容易很多。

## 给团队管理员

如果你负责给同事提供可用环境，请至少确认：

- Deployer 首页可打开。
- CodeTalk 前端可打开。
- Backend `/health` 返回 `status=ok`。
- 可以创建工作空间。
- 设置页可以加载 LLM 配置。
- 停止服务后端口会释放。
- 如果提供增强组件，组件的路径、端口和 health/status 都已验证。

内部开发和测试团队的 release gate、E2E、blocker 判定，请看：

```text
docs/INTERNAL_RELEASE.md
```
